# Отчет по Лабораторной работе №5

---

## 1. Введение и концепция обеспечения качества
В рамках данной лабораторной работы был реализован переход от этапа активного прототипирования к этапу стабилизации и обеспечения промышленного качества программного продукта. Согласно современным методологиям (Agile, XP), качество не является отдельной фазой в конце цикла, а интегрируется в процесс разработки через автоматизацию и непрерывный рефакторинг.

**Цели работы:**
1.  **Архитектурная декомпозиция:** Разделение монолитного кода на независимые модули для повышения показателя поддерживаемости (Maintainability).
2.  **Верификация алгоритмов:** Создание системы автоматизированного контроля математических вычислений.
3.  **Инкапсуляция бизнес-логики:** Использование принципа единственной ответственности (SRP) из набора SOLID.
4.  **Внедрение Bug Tracking системы:** Систематизация процесса обнаружения и исправления дефектов через GitHub Issues.

## 2. Глубокий рефакторинг архитектуры

### 2.1. Анализ "Технического долга"
На этапах ЛР №1–4 код приложения имел высокую степень зацепления (High Coupling). Математические операции (суммирование баланса, вычисление процентов накопления) были жестко вшиты в функции отрисовки DOM. Это делало невозможным проведение Unit-тестов без симуляции всей среды браузера и подключения к облачной БД Firebase.

### 2.2. Создание модуля бизнес-логики (`utils.js`)
Для решения проблемы был выделен независимый логический слой. В файл `utils.js` были перенесены все вычислительные мощности приложения.
*   **Используемый подход:** Чистые функции (Pure Functions). Функции не изменяют внешнее состояние, не обращаются к глобальным переменным и возвращают предсказуемый результат для одних и тех же входных данных.
*   **Экспортируемый объект `financeUtils`:** Включает методы `calculateTotals` и `calculateGoalProgress`.

### 2.3. Интеграция в основной цикл приложения
Файл `app.js` был подвергнут глубокой модификации. Все вызовы математики теперь делегируются модулю `utils.js`. Это позволило сократить объем `app.js` на 15% и сделать его код чисто декларативным (описывающим *что* отображать, а не *как* считать).

## 3. Система автоматизированного тестирования (Unit Testing)

### 3.1. Стратегия верификации
Выбрана стратегия «черного ящика» для отдельных функций. Мы подаем на вход функции заранее подготовленные структуры данных (Mocks) и сравниваем результат с эталонным.

### 3.2. Тестовое покрытие (Test Suite)
В разработанном тестовом драйвере `test.js` реализованы следующие сценарии:

*   **Сценарий №1: Агрегация транзакций.**
    *   *Вход:* Массив из 10 объектов (разные типы: `income`, `expense`).
    *   *Ожидание:* Корректная сумма доходов, расходов и дельты (баланса).
    *   *Критичность:* Высокая (основа финансовой достоверности).

*   **Сценарий №2: Вычисление прогресса финансовой цели.**
    *   *Вход:* Текущий баланс (500) и целевая сумма (2000).
    *   *Ожидание:* Значение `25.0`.
    *   *Проверка точности:* Использование чисел с плавающей запятой (проверка корректности округления).

*   **Сценарий №3: Обработка граничных условий (Edge Cases).**
    *   *Вход:* Целевая сумма равна `0` или баланс отрицателен.
    *   *Ожидание:* Функция не должна выбрасывать исключение (Runtime Error). Возврат `0`.
    *   *Цель:* Обеспечение отказоустойчивости приложения.

### 3.3. Результаты прогона тестов
Тестирование проводилось во встроенной консоли разработчика. Все 100% тестов прошли успешно (Status: SUCCESS). Логика расчетов признана стабильной и готовой к масштабированию.

## 4. Управление дефектами (Bug Tracking & QA)

### 4.1. Жизненный цикл дефекта в проекте
В ходе работы над качеством был внедрен стандартный флоу обработки ошибок в GitHub Projects:
1.  **Backlog:** Регистрация бага пользователем/тестировщиком.
2.  **In Progress:** Анализ кода и написание исправляющего патча.
3.  **Review:** Проверка исправления вторым участником команды (Парное программирование).
4.  **Done:** Закрытие задачи с фиксацией версии.

### 4.2. Анализ исправленного дефекта #5
**Тип:** Логическая ошибка / Ошибка отображения (UI/UX Bug).
**Симптомы:** При определенных значениях баланса (например, доход 100 при цели 300) прогресс-бар отображал `33.333333333333336%`.
**Причина:** Отсутствие явного приведения типов и форматирования при выводе результата деления в JavaScript.
**Метод устранения:** В модуле `utils.js` внедрен метод `toFixed(1)` с последующим преобразованием в числовой формат через `parseFloat`. Это обеспечило строгое соответствие дизайн-макету.
