**Отчет по лабораторной работе №4**

**Тема:** Исследование стиля «Парное программирование»

1. **Какую задачу реализовывали?**
Мы реализовали модуль «Финансовые цели» (Savings Goals). Задача включала:
- Создание новой коллекции в Firebase Firestore для хранения целей.
- Верстку прогресс-баров и модального окна добавления цели.
- Логику расчета прогресса (в %): соотношение текущего баланса пользователя к сумме поставленной цели.

2. **Какой стиль показался более приемлемым? Почему?**
Мы попробовали два стиля:
- **Водитель-Штурман:** Саша писал код, Илья проверял логику на лету и следил за именованием переменных. Это помогло избежать багов в расчете процентов.
- **Пинг-Понг:** Один писал структуру функции в Firebase, второй — её реализацию.
**Итог:** Стиль «Водитель-Штурман» показался более эффективным для этой задачи, так как он позволял не прерывать "поток" написания кода и сразу обсуждать архитектурные решения.

3. **Что получилось? Положительные впечатления.**
- Скорость поиска опечаток увеличилась: пока «водитель» пишет длинную строку, «штурман» уже видит пропущенную скобку.
- Код стал чище, так как в процессе обсуждения мы сразу проводили рефакторинг.
- Оба участника команды теперь на 100% знают, как работает модуль целей.

4. **Что не получилось? Как можно это поправить?**
- Иногда возникала задержка связи (latency) при удаленной работе через Code With Me, из-за чего курсоры "прыгали".
- Было сложно не перебивать друг друга в начале.
- **Как исправить:** Использовать голосовую связь с минимальной задержкой (Discord) и четко регламентировать время смены ролей (например, каждые 20 минут).

5. **Будете ли использовать ПП для других задач? Каких?**
Да, ПП идеально подходит для критически важных или сложных задач, таких как:
- Проектирование схемы базы данных.
- Реализация сложной аналитики или фильтрации.
- Отладка (дебаг) трудновоспроизводимых ошибок.
